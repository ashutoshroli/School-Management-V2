# Trial / Demo Deployment Guide

This deploys a **free-tier trial** of the app so you can click around a
live URL - it is **not** a production setup (see caveats at the bottom).

Stack used (all have free tiers, no credit card needed for the DB/backend):

| Piece | Service | Why |
|---|---|---|
| Database | [Neon](https://neon.tech) (Postgres) | Generous free tier, instant provisioning, gives you a `DATABASE_URL` in ~30 seconds |
| Backend API | [Render](https://render.com) | Free web service tier, reads `render.yaml` in this repo automatically |
| Frontend | [Vercel](https://vercel.com) | Purpose-built for Next.js, free tier, auto-detects the framework |

Total time: ~15-20 minutes, mostly waiting for builds.

---

## Step 1 - Push this repo to your own GitHub account

All three services deploy by connecting to a GitHub repo. If this code
isn't already in a repo under **your** GitHub account, push it there
first (fork or push the `main` branch).

## Step 2 - Create the database (Neon)

1. Go to https://neon.tech and sign up (GitHub login works).
2. Create a new project (any region, any name - e.g. `school-erp-trial`).
3. On the project dashboard, copy the **connection string**. It looks like:
   ```
   postgresql://neondb_owner:xxxxx@ep-xxxx-xxxx.us-east-2.aws.neon.tech/neondb?sslmode=require
   ```
4. Keep this tab open - you'll paste this as `DATABASE_URL` in Step 3.

## Step 3 - Deploy the backend (Render)

1. Go to https://render.com and sign up / log in.
2. Click **New > Blueprint**, connect your GitHub account, and select
   this repository. Render will detect `render.yaml` at the repo root
   and propose a service called `school-erp-backend`.
3. Before clicking "Apply", you'll be prompted for the env vars marked
   `sync: false` in `render.yaml`. Fill in:

   | Env var | Value |
   |---|---|
   | `DATABASE_URL` | The Neon connection string from Step 2 |
   | `FRONTEND_URL` | Leave as a placeholder for now (e.g. `https://placeholder.vercel.app`) - you'll update it in Step 5 once you know the real Vercel URL |
   | `GOOGLE_CLIENT_ID` | Leave **blank** for a trial - Google sign-in is optional, email/password login works without it |
   | `GOOGLE_CLIENT_SECRET` | Leave **blank** |
   | `GOOGLE_CALLBACK_URL` | Leave **blank** |

   `JWT_SECRET` is auto-generated by Render (`generateValue: true`) -
   you don't need to fill it in.

4. Click **Apply**. Render will run `backend/scripts/build.sh`, which:
   - installs the `db` package and generates the Prisma client
   - installs backend dependencies
   - runs `prisma db push` to create all tables in your Neon database
     (this repo doesn't commit migration files, so this is the trial
     equivalent of running migrations)
   - builds the TypeScript backend
5. Once deployed, copy the backend's public URL, e.g.
   `https://school-erp-backend-xxxx.onrender.com`. Test it:
   ```
   https://school-erp-backend-xxxx.onrender.com/api/health
   ```
   should return `{"success":true,"message":"...running..."}`.

   > Free Render web services **sleep after ~15 minutes of inactivity**
   > and take ~30-50 seconds to wake up on the next request - this is
   > expected on the free tier, not a bug.

## Step 4 - Seed demo data (one-time)

The database is empty after `db push` - it created the tables but no
rows. Run the seed script once, from your own machine, pointed at the
Neon database:

```bash
cd db
DATABASE_URL="<your Neon connection string>" npm install
DATABASE_URL="<your Neon connection string>" npx prisma generate
DATABASE_URL="<your Neon connection string>" npm run seed
```

This creates a demo organization/branch/classes/subjects/chart of
accounts, and two login accounts (also printed at the end of the seed
script):

| Role | Email | Password |
|---|---|---|
| Super Admin | `superadmin@abcschool.edu.in` | `Admin@123` |
| Branch Admin | `branchadmin@abcschool.edu.in` | `Admin@123` |

**Change these passwords (or delete these accounts) before sharing the
trial URL with anyone outside your team.**

## Step 5 - Deploy the frontend (Vercel)

1. Go to https://vercel.com and sign up / log in.
2. Click **Add New > Project**, import this same GitHub repo.
3. Vercel will ask for the project **Root Directory** - set it to
   `frontend` (important: this is a monorepo, the Next.js app is not at
   the repo root).
4. Under **Environment Variables**, add:

   | Name | Value |
   |---|---|
   | `NEXT_PUBLIC_API_URL` | `https://school-erp-backend-xxxx.onrender.com/api` (your Render URL from Step 3, with `/api` appended) |
   | `NEXT_PUBLIC_APP_NAME` | `School ERP` |

5. Click **Deploy**. Once done, copy the frontend's URL, e.g.
   `https://school-erp-trial.vercel.app`.

## Step 6 - Connect the two (fix CORS)

Go back to **Render > school-erp-backend > Environment**, and update:

| Env var | Value |
|---|---|
| `FRONTEND_URL` | Your real Vercel URL from Step 5, e.g. `https://school-erp-trial.vercel.app` |

Save - Render will redeploy automatically. This matters because the
backend's CORS policy only allows requests from `FRONTEND_URL`
(`backend/src/app.ts`); until this is set correctly, the frontend will
be able to load but every API call will fail with a CORS error in the
browser console.

## Step 7 - Log in and try it

Open your Vercel URL and log in with the Super Admin credentials from
Step 4. From there you can explore students, fees, accounting, HR,
etc. with the seeded demo data.

## Known trial-only limitations

These are all intentional simplifications for a quick trial, not bugs:

- **No file storage persistence**: uploaded documents/avatars are saved
  to local disk on the Render container (`backend/src/services/storage.service.ts`).
  Render's free tier has an ephemeral filesystem - uploaded files will
  be lost on every redeploy/restart. Fine for clicking around, not for
  real use. A real deployment should swap in an S3/GCS-backed
  `StorageProvider`.
- **Google OAuth is disabled** unless you fill in `GOOGLE_CLIENT_ID`/`GOOGLE_CLIENT_SECRET` -
  email/password login (used by all staff roles) works regardless.
- **Online payments (Razorpay) won't work** unless you also set
  `RAZORPAY_KEY_ID`/`RAZORPAY_KEY_SECRET` (and ideally use Razorpay's
  test-mode keys, not live ones, for a trial).
- **Email notifications are silently skipped** unless `SMTP_HOST`/`SMTP_USER`/`SMTP_PASS`
  are set - everything else in the app still works, payments/actions
  just won't send a confirmation email.
- **SMS/WhatsApp notifications are stubs** regardless of environment -
  they log instead of sending (see README's "Known limitations" section).
- **Cold starts**: Render's free tier sleeps the backend after 15
  minutes idle; the first request after that takes ~30-50 seconds.
- **No committed migration history**: this trial setup uses
  `prisma db push` instead of `prisma migrate deploy` since no
  migrations are committed to the repo (see `.gitignore`). A real
  production setup should generate and commit proper Prisma migrations.
